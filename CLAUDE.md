# HYROX Bordeaux — Training Tracker

Maintenance guide for this project. Read this before editing.

## What it is

A single-file web app to follow a 14-week HYROX preparation program (race: **2026-10-03**,
HYROX Bordeaux Solo Open). It displays the program, lets the user log each session, and
shows progress. French UI. No build step, no framework, no dependencies.

- **Live:** https://paulgrvd.github.io/hyrox-prog/
- **Repo:** https://github.com/PaulGrvd/hyrox-prog (GitHub Pages, branch `main`, path `/`)
- **Deploy:** push to `main` → Pages rebuilds automatically (~1–3 min, CDN-cached).

## Files

| File | Role |
|------|------|
| `index.html` | The entire app — HTML + CSS + JS + the embedded program data. Edit this. |
| `prepa_hyrox_bordeaux_solo_open.csv` | **Source of truth** for the program. Edited by the user. |
| `scripts/refresh-data.js` | Syncs the CSV into `index.html`. Run after any CSV change. |
| `CLAUDE.md` | This file. |

## Critical invariants — do not break these

1. **`STORAGE_KEY` must stay `'hyrox-bordeaux-2026'`.** It keys the user's tracking data in
   `localStorage`. Renaming it = the user silently loses all logged data.
2. **Keep the `log` entry shape backward-compatible.** Old saved entries must still render.
   Add fields, don't rename/remove existing ones.
3. **Data stays embedded, not fetched.** The page must work from `file://` (double-click) and
   from Pages. Do not switch to `fetch('...csv')` — it breaks the local-file use case.
4. **The program data is embedded** in a JS template literal, so editing the CSV does NOT update
   the page by itself. You must run `scripts/refresh-data.js` (see below).

## Data flow

```
prepa_hyrox_bordeaux_solo_open.csv
   │  (scripts/refresh-data.js: escapes \ ` ${ and replaces the block)
   ▼
index.html:  const CSV = ` ...19 cols, 98 session rows... `;
   │  parseCSV()  → rows
   ▼
const sessions = [{ id(=date), idx, date, jour, semaine, bloc, focus,
                    type, seance, duree, rpe, priorite, notes }, ...]
```

CSV columns (19): `Date, Jour, Semaine, Bloc, Focus semaine, Type séance, Séance prévue,
Durée prévue, RPE cible, Priorité, Fait ?, Durée réelle, RPE réel, Sommeil /10,
Fatigue jambes /10, Douleur /10, Golf joué ?, Score ou ressenti golf, Notes / splits`.
The page reads only the *planned* columns; the tracking columns are filled by the user in-app
(stored in localStorage, not the CSV).

## localStorage schema

Key `hyrox-bordeaux-2026` → JSON object `log`, keyed by session date (`id`):

```js
log["2026-06-29"] = {
  done: true,                 // session marked complete
  duree: "62", rpe: "8",      // actual values (strings from inputs)
  sommeil: "7", fatigue: "5", douleur: "1",
  golf: "Oui", golfScore: "…", userNotes: "…",
  steps: { 0:true, 2:true }   // per-exercise checkboxes, index → bool
}
```
Only logged sessions appear in `log`. `entry(id)` lazily creates the object on write.

## index.html structure

Section markers exist as comments in both the `<style>` and `<script>`. Jump by them.

**CSS sections:** Header · Stats · Controls · Blocks & weeks · Session cards ·
Tracking form (shared) · Session detail page.

**JS sections / key functions:**
- `Raw program data` — `const CSV = ...` (regenerated by the refresh script).
- `CSV parser` — `parseCSV()` (handles quoted fields with commas/newlines).
- `sessions` array build (maps rows → objects by header name).
- `Type styling` — `TYPE_STYLE` map + `tstyle()` (color/icon per session type).
- `HTML escaping` — `esc()`. **Use it on any CSV/user text injected into HTML**
  (session text contains `<`, e.g. `transitions <45s`, which breaks HTML if unescaped).
- `Parse "Séance prévue"` — `METRIC_RX`, `parseStep()`, `parseSession()`, `prettyChip()`.
  Turns the long session string into structured exercises: splits on ` + ` and sentence
  breaks; pulls out a `Technique:` cue (capital T only); extracts metric chips
  (sets×reps, `@kg`, distances, durations, RPE). Heuristic — see "Updating the parser".
- `Render program` (list view) — `render()` → `buildGrouped()` (groups by bloc → semaine)
  → `cardHTML()` (one card per session).
- `Session detail page` — `renderDetail()` (full session page) + `exHTML()` (one exercise row)
  + `attachDetailHandlers()` (back/prev/next nav, done toggle, per-exercise checkboxes, form inputs).
- `Router` — `route()`, hash-based: `#s/<date>` shows the detail page, empty hash shows the list.
  Toggles `#listView` / `#detailView`. Back button works via `hashchange`.
- `Filter UI` — status / type / bloc filters.
- `Export` — CSV export of tracking.
- `Backup & restore` — `downloadFile()`; JSON backup (full `log`, incl. `steps`) and restore
  (merge: imported dates overwrite, others kept). This is the only recovery path — keep it working.
- `Init` — `populateFilters(); renderCountdown(); route();`

Views: everything lives in `#listView`; the session page renders into `#detailView`. Only one
is visible at a time (controlled by `route()`).

## Common tasks

### Update the program (CSV changed)
```bash
node scripts/refresh-data.js      # re-embeds CSV, validates 19 cols / rows
# then verify + deploy (below)
```

### Edit the app (HTML/CSS/JS)
Edit `index.html` directly. Respect the invariants above. Reuse `esc()` for injected text.

### Verify before deploying (no browser needed)
```bash
# 1. Script syntax
node -e 'const fs=require("fs");const h=fs.readFileSync("index.html","utf8");require("fs").writeFileSync("/tmp/s.js",h.match(/<script>([\s\S]*?)<\/script>/)[1]);' && node --check /tmp/s.js && echo OK
```
A headless browser (gstack `/browse` or Playwright) is the ideal check but was not installed
in past sessions; the Node syntax check + manual reload has been the practical fallback. There
are no committed tests — prior validation used throwaway scripts (parser unit test + a stub-DOM
render smoke test over all 98 sessions). Recreate them in a scratch dir if making risky changes.

### Deploy
```bash
git add -A
git commit -m "..."   # end commits with the Co-Authored-By trailer
git push origin main  # Pages auto-builds; confirm with:
curl -s "https://paulgrvd.github.io/hyrox-prog/?cb=$(date +%s)" | grep -o '<title>[^<]*</title>'
```
Pages CDN caches a few minutes; use a `?cb=` cache-buster when checking for a fresh change.

## Updating the parser (`parseSession`/`parseStep`)

It's a heuristic over French free-text, so it can mis-split unusual phrasing. When the source
text changes shape, re-test against representative rows of each `Type séance`
(Force jambes, Running seuil / intervalles, Force haut + stations, HYROX spécifique,
Zone 2, Golf / récup active, Golf long / repos, RACE). Watch for:
- the `Technique:` cue regex (must match capital `Technique`, not incidental lowercase
  "technique" inside e.g. "HYROX technique 4 tours:");
- sentence-split eating decimals/weights (it only splits on `. ` before a capital/digit);
- empty exercise names (parser falls back to the label or raw text — there should be none).

## Notes / open items

- **No cross-device sync.** Tracking is per-browser (localStorage). Moving data between phone
  and laptop is manual via Backup/Restore. A real sync would need a backend (not built).
- **Public repo** (required for free Pages). Content is just the schedule; no personal data,
  and tracking is never committed. Switching to private needs GitHub Pro or another host.
- **iOS:** advise the user to open via "Add to Home Screen" (avoids Safari's ~7-day storage
  eviction and storage split between Safari and the standalone app).
- iOS meta tags (`apple-mobile-web-app-*`, `theme-color`, `viewport-fit=cover`) are set in `<head>`.
